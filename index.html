Borla Tracker WMS

A digital transformation initiative designed to bridge the gap between waste producers and collectors in Ghana.

Executive Summary: The project delivers a multi-platform ecosystem: a Flutter-based mobile application for clients and collectors, and a React + TypeScript web dashboard for operational oversight. Backed by a robust Django/PostGIS API, the system automates assignment, standardizes pricing, and digitizes payments.

Introduction & Problem Context

Ghana‚Äôs waste management sector struggles with rapid urbanization, generating approximately 12,710 tonnes of solid waste daily, with a collection deficit of nearly 40%.

The Problem

The absence of a unified digital platform connecting households to waste collectors results in:

  • Operational Invisibility: Supervisors cannot track where collectors are in real-time.
  • Revenue Leakage: Cash-based transactions result in financial discrepancies.
  • Service Inequity: Low-income zones suffer from irregular pickups due to lack of data.

Project Objectives

Digital Integration

Create a seamless interface for Clients to request services and Collectors to receive geo-optimized tasks.

Standardization

Implement an automated pricing engine based on waste type (Hazardous, Plastic, General).

Optimization

Reduce operational coordination overhead by 40% through automated dispatching.

System Requirements (SRS)

Functional Modules

üîê Auth & Security

  • JWT: Secure, stateless authentication.
  • RBAC: Strict role separation for API endpoints.

üöõ Operations

  • Geospatial Assignment: Assign requests to nearest collector.
  • Evidence Locking: Must upload "Before/After" photos to complete.
  • Proximity: Must be within 100m to close ticket.

üí≥ Payments

  • Gateways: Mobile Money & Card (Paystack).
  • Splitting: Auto-split revenue between Platform, Company, and Collector.

Non-Functional Requirements

  • Performance: API response < 200ms.
  • Availability: 99.5% uptime target.
  • Scalability: Kubernetes scaling for 10,000+ users.
  • Data Integrity: Automated PostgreSQL backups.

Architecture & Technology Stack

The system employs a Microservices-ready Layered Architecture.

Tech Stack

Flutter (Mobile) React + TypeScript (Web) Django REST Framework PostgreSQL + PostGIS Redis + Celery

Why this stack?

Mobile (Flutter): Compiles to native ARM code for superior performance in GPS-heavy tasks compared to hybrid wrappers.

Web (React + TS): Component-based structure ideal for complex dashboards with Type Safety.

Backend (Django + PostGIS): Industry standard for handling complex geospatial queries (e.g., "Find points inside this polygon").

Business Logic Implementation

Core algorithmic logic driving the backend operations.

Pricing Algorithm

The system calculates service fees dynamically based on waste volume, type, and zone density.

Final Price = (Base Rate √ó Bag Count) √ó Waste Type Multiplier √ó Zone Factor

Multipliers: Hazardous (2.0x), Recyclables (0.8x), General (1.0x).
Zone Factor: High-density (1.2x), Low-density (1.0x).

Incentive & Penalty

Incentive: Base + Compliance Bonus - Penalty.
Penalties: Applied for GPS mismatch (>100m) or segregation < 70%.

Route Optimization

Uses Nearest Neighbor Heuristic with time slot constraints to group requests by geographic zone.
