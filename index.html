Borla Tracker WMS

A digital transformation initiative designed to bridge the gap between waste producers and collectors in Ghana.

Executive Summary: The project delivers a multi-platform ecosystem: a Flutter-based mobile application for clients and collectors, and a React + TypeScript web dashboard for operational oversight. Backed by a robust Django/PostGIS API, the system automates assignment, standardizes pricing, and digitizes payments. The solution targets a 95% on-time pickup rate and aims to integrate the informal sector (freelance riders) into a formal economy.

Introduction & Problem Context

Ghanaâ€™s waste management sector struggles with rapid urbanization, generating approximately 12,710 tonnes of solid waste daily, with a collection deficit of nearly 40%. In cities like Accra and Kumasi, the "Communal Container Collection" (CCC) system is often overwhelmed. Private sector involvement, pioneered by Zoomlion, utilizes tricycle operators to navigate narrow streets, but these operations lack central coordination.

The Problem

The absence of a unified digital platform connecting households to waste collectors results in inefficient routing, opaque pricing, and a lack of accountability.

  • Operational Invisibility: Supervisors cannot track where collectors are in real-time.
  • Revenue Leakage: Cash-based transactions result in financial discrepancies.
  • Service Inequity: Low-income zones suffer from irregular pickups due to lack of data-driven scheduling.

Project Objectives

Digital Integration

To create a seamless interface for Clients to request services and Collectors to receive geo-optimized tasks.

Standardization

To implement an automated pricing engine that calculates fees based on waste type (Hazardous, Plastic, General) and zone density.

Optimization

To reduce operational coordination overhead by 40% through automated dispatching.

Formalization

To provide a mechanism for independent tricycle operators to be recruited and verified by licensed companies.

Research & Methodology

3.1 Research Approach

The team adopted a User-Centric Design approach, utilizing:

  • Stakeholder Interviews: Engaged with local tricycle riders to understand their navigation and payment pain points.
  • Competitor Analysis: Evaluated existing manual dispatch methods used by local assemblies.

3.2 Root Cause Analysis

Root Cause 1

Manual Dispatching: Supervisors rely on phone calls to assign tasks, leading to delays.

Root Cause 2

Lack of Trust: Clients are hesitant to pay upfront due to a history of missed pickups.

Root Cause 3

Fragmented Workforce: Thousands of willing collectors exist but have no platform to advertise their availability.

System Requirements Specification (SRS)

Functional requirements broken down by module.

Module A: Authentication & Security

  • FR-AUTH-01: System must use JWT (JSON Web Tokens) for secure, stateless authentication across mobile and web.
  • FR-AUTH-02: Role-Based Access Control (RBAC) must strictly segregate Client, Collector, and Admin API endpoints.

Module B: Collector Recruitment (Discovery)

  • FR-DISC-01: Independent collectors must be able to search for registered companies within their GPS zone.
  • FR-DISC-02: The mobile app must allow document uploads (Driverâ€™s License, Vehicle Registration) for vetting.
  • FR-DISC-03: Company Admins must have a "Talent Pool" interface to approve/reject applications and onboard drivers.

Module C: Service Operations

  • FR-OPS-01: Geospatial Assignment: The backend must assign requests to the nearest available collector within the same geofence.
  • FR-OPS-02: Evidence Locking: Tasks cannot be marked "Complete" without uploading "Before" and "After" images.
  • FR-OPS-03: Proximity Validation: Collectors must be within 100 meters of the pickup coordinates to close a ticket.

Module D: Payments

  • FR-PAY-01: Integration with Mobile Money (MTN/Telecel) and Credit Cards via a secure gateway (e.g., Paystack).
  • FR-PAY-02: Automated revenue splitting between the Platform, Company, and Collector.

Non-Functional Requirements

  • Performance: The API must respond to task queries in <200ms to ensure fluidity on 3G networks.
  • Availability: 99.5% uptime target to support daily operations.
  • Scalability: Horizontal scaling via Kubernetes to support up to 10,000 concurrent users.
  • Data Integrity: Daily automated backups of PostgreSQL databases.

System Architecture & Technology Stack

6.1 Architectural Design

The system employs a Microservices-ready Layered Architecture:

Presentation Layer

Cross-platform Mobile Apps and a Responsive Web Dashboard.

Application Layer

A Django REST Framework (DRF) API Gateway handling business logic, with Celery workers for background tasks (e.g., generating weekly schedules).

Data Layer

PostgreSQL (Relational Data) + PostGIS (Spatial Data) + Redis (Caching).

6.2 Tech Stack Justification

  • Mobile (Flutter): Chosen for its ability to compile to native ARM code for both iOS and Android from a single codebase, offering superior performance for GPS-heavy tasks compared to hybrid wrappers.
  • Web (React + TypeScript): React provides a component-based structure ideal for complex dashboards, while TypeScript ensures type safety, reducing runtime errors in the admin portal.
  • Backend (Django + PostGIS): Django offers rapid development with built-in security. PostGIS is the industry standard for handling complex geospatial queries (e.g., "Find points inside this polygon").

System Integration & Workflow

How the Borla Tracker ecosystem functions as a unified platform to automate waste collection.

1

Request Initiation (Client App)

A household client logs into the Flutter Mobile App and schedules a pickup. They select the waste type (e.g., Plastic), bin size, and date. This data is sent securely to the Django Backend via REST API.

2

Intelligent Dispatch (Backend Logic)

The Backend API receives the request. Using PostGIS, it identifies the client's geospatial zone and automatically assigns the task to the nearest available collector operating within that polygon.

3

Field Execution (Collector App)

The assigned driver receives a push notification on their Collector App. They accept the job and use the in-app Geospatial Navigation to reach the client. Upon arrival, they must upload "Before" and "After" photos to lock the evidence.

4

Monitoring & Payment (Web Dashboard)

The Supervisor watches this progress in real-time on the React Web Dashboard map. Once the job is marked complete and verified, the payment is processed, and revenue is split automatically.
